## 一、当前用户旅程 —— 逐屏解剖

### 用户看到的
```
┌──────────────────────────────────────────────────────────┐
│  Nav: 首页 | 品牌资产 | ...                    [中|EN]   │
```

### 反直觉洞察 #1：页面在"请求决策"而非"引导行动"

这个页面的认知负荷极高。用户在 3 秒内需要处理：时长（5 档）、场景（3 种）、模式（2 种）、产品输入（3 个字段）、API 密钥（4 个）、素材上传（可选）、右侧场景说明——约 15 个独立决策点。

乔布斯的 iPod 发布会只问用户一个问题："你想把 1000 首歌放进口袋吗？"

我们的页面问了 15 个问题。

**推荐：** 将页面重构为"向导式"两步流。第 1 步只需要场景 + 产品名（2 个输入），第 2 步展示 AI 推荐的所有配置（时长、平台、策略），用户确认或微调。API 密钥移到设置页，素材上传作为高级功能折叠。

## 二、业务链路 —— 用户真正关心的 vs 我们构建的

### 用户心智模型
```
输入产品 → 获得视频 → 发布到平台
```

### 我们构建的模型
```
输入产品 → [strategy → scripts → compliance → storyboards → keyframe_images 
→ video_prompts → thumbnail_prompts → seedance_clips → tts_audio 
→ thumbnail_images → assemble_final → audit] → 获得结果
```

### 反直觉洞察 #2：逐步模式是开发者工具，不是产品功能

我们在"逐步模式"上投入了大量工程资源——Step API、下行清空、编辑重跑。但一个内容创作者永远不会想手动执行 12 个步骤。这就像是给用户一个"手动挡"的 iPhone。

**数据点：** 12 个步骤中，用户真正能理解和操作的只有 3 个：
- "撰写视频脚本"（scripts）—— 用户理解
- "生成视频提示词"（video_prompts）—— 用户可能理解
- "合成最终成片"（assemble_final）—— 用户理解

其余 9 个步骤（strategy, compliance, storyboards, keyframe_images, thumbnail_prompts, seedance_clips, tts_audio, thumbnail_images, audit）是**工程概念**，对用户无意义。

**推荐：** 保留逐步模式的 API 能力（对调试有价值），但用户界面重新设计为 3 步向导：**写文案 → 生成画面 → 合成导出**。每步内部自动执行多个 pipeline 步骤，但用户只看到 3 个阶段。

## 三、反直觉洞察全集

### #1：功能越多，产品越弱
当前产品形态：12 步管线、15 个 Skill、5 档时长、3 种场景、2 种模式、双语切换、3 个管理页面、44 个 API 端点。功能矩阵看起来很美。

但是——**没有一条从"打开浏览器"到"拿到第一条可发布视频"的完整路径被验证过。** 所有功能都是假设性的，因为端到端链路从未跑通。

这就像建了一栋 12 层的酒店，但大堂到房间的电梯还没装。客人在楼下看到宏伟的建筑，但上不去。

### #2：质量门控制造焦虑而非行动力
`QualityDashboard` 展示 PASS/WARN/FAIL 徽章。但如果用户看到 "Face Consistency: FAIL"——他们能做什么？

没有任何产品机制让用户**修复**质量问题。重新生成？编辑输入？选择不同的关键帧？这些都是"可做"但没有**引导**的动作。

苹果的设计原则：**只展示可操作的信息。** 如果用户不能基于这个信息采取行动，就不要展示。

**推荐：** 质量门控结果**不直接展示给用户**。改为后台自动重试机制：不达标 → 自动重新生成 → 3 次后选最优结果 → 仅当全部失败时才向用户展示"生成遇到问题，已自动重试 3 次，是否更换产品描述重试？"

### #3：英文母语是正确的方向，但产品心智模型仍然是中文
UI 可以切换中英文，但：
- 默认产品名 "孕妇枕" 是中文
- 默认品牌 "Momcozy" 是英文品牌名但产品场景是中文语境
- 场景描述中"网红二创带链接"的逻辑是中国电商生态（抖音+小红书）
- 翻译层只处理产品名/USP，不处理品牌调性、文化语境

**推荐：** 创建两套完整的场景模板——一套中文（中国电商生态）、一套英文（Amazon/Shopify 生态）。语言切换时同步切换场景预设。

### #4：最大的上线阻碍不是技术，是"首体验"
假设明天一个真实用户打开产品。他输入 "baby monitor"，点击生成。3-5 分钟后，他得到什么？

- **最好情况：** 一条 30s 英文产品视频，画面连贯，配音清晰，可以发布
- **大概率：** 部分步骤 stub，视频只有 5s 纯色画面，音频静音，缩略图是占位符
- **最差情况：** Pipeline 中途崩溃，返回 500 错误

**三种结果对用户留存的影响天差地别。** 在首体验没验证之前，所有功能迭代都是沙上建塔。

### #5：SceneSelector 的三个场景有根本性的产品定位问题
- **"网红二创带链接"** —— 需要输入视频 URL，但我们没有视频下载能力验证过
- **"品牌宣传片"** —— 需要品牌资产包（logo、色板、字体），品牌资产上传页面做了但资产库是空的
- **"商品直拍"** —— 只需要产品名和卖点，这是唯一真正"可以工作"的场景

**推荐：** 产品首发只开放**"商品直拍"一个场景**。把 100% 的体验打磨集中在一个场景上，让它从"能用"变成"好用"。网红二创和品牌宣传作为 "Coming Soon" 展示，收集用户意向后再开发。

## 四、上线差距量化

| 维度 | 当前状态 | 上线要求 | 差距 |
|------|---------|---------|------|
| **端到端产出** | 从未生成过真实视频 | 3 条不同产品的可发布视频 | 🔴 未验证 |
| **首体验时间** | 5-8 分钟（含加载等待） | < 2 分钟 | 🔴 |
| **成功率** | 未知（未测试） | > 80% pipeline 完成率 | 🔴 |
| **视频质量** | 未知（stub 占多数） | 人工可接受率 > 70% | 🔴 |
| **错误恢复** | 无用户可操作的恢复路径 | 失败时引导用户下一步 | 🔴 |
| **前端性能** | 开发模式，未构建 | 生产构建 + 静态导出 | 🟡 |
| **认证** | 无 | JWT + API Key | 🔴 |
| **多租户** | 无 | 租户隔离 | 🔴 |
| **分发** | Mock | TikTok/Shopify 真实发布 1 条 | 🔴 |
| **文档** | API 参考 + 快速指南 | 用户操作手册 + 常见问题 | 🟡 |
| **自测覆盖** | 0 条端到端 | 3 个场景各 2 条 | 🔴 |

**结论：距离可上线（给第一个外部客户试用）还有 3-4 周。** 当务之急不是加功能，而是把 "商品直拍" 场景从 🔴 变成 🟢。

## 五、战略调整建议

### 黄金路径：商品直拍首发

```
砍掉：网红二创（S3）、品牌宣传（S2合并但暂不对外）
聚焦：商品直拍（S1）—— 唯一可以端到端跑通的场景
资源：100% 精力打磨一条路径
目标：2 周内产出 5 条可发布的商品视频，3 个不同品类
```

### 产品界面重设计（5 个关键改动）

**1. 首页改为单输入框**
```
┌─────────────────────────────────────────┐
│                                         │
│     Create Product Video with AI        │
│                                         │
│   ┌─────────────────────────────────┐   │
│   │  Enter product name or URL...   │   │
│   └─────────────────────────────────┘   │
│                                         │
│   [Generate Video →]                    │
│                                         │
│   ── or ──                              │
│   Advanced: Duration · Platform · Brand │
└─────────────────────────────────────────┘
```

**2. 生成过程展示 3 阶段而非 12 步**
```
[写文案 ████████████ ✅] → [生成画面 ██████░░░░ 🔄] → [合成导出 ░░░░░░░░░░ ⏳]
```

**3. 结果页以视频预览为核心，而非 JSON 数据**
- 顶部：视频播放器（自动播放、静音、全屏）
- 下方：3 个 tab（Script / Thumbnails / Technical Details）
- 右上角：Download · Share · New

**4. 质量信息降级为后台自动处理**
- 用户看不到 PASS/WARN/FAIL
- 低质量结果自动重新生成（最多 3 次）
- 仅全部失败时才提示

**5. 语言切换联动场景模板**
- EN 模式：默认 Amazon/Shopify 生态，产品名 placeholder="e.g. Maternity Pillow"
- 中文模式：默认抖音/小红书生态，产品名 placeholder="例如：孕妇枕"

## 六、需要你的决策

以下 4 个决策点需要你拍板，我带着推荐和理由：

### 决策 1：是否暂时隐藏 S3（网红二创）和品牌宣传入口？
**推荐：** 是。UI 上折叠为 "More Scenarios (Coming Soon)"，只保留商品直拍作为主动作入口。
**理由：** 当前 S3 的视频下载和 S2 的品牌资产都未验证过，展示但不工作比不展示更伤害信任。

### 决策 2：是否将 12 步逐步模式从用户界面移除，改为开发者调试入口？
**推荐：** 是。逐步模式改为 URL 参数激活（`?debug=true`），对普通用户不可见。用户界面改为 3 阶段进度条。
**理由：** 12 步的认知负荷过高，降低首体验摩擦。逐步 API 保留，调试时可通过 URL 参数启用。

### 决策 3：视频时长选择——前置（当前方案）还是后置（AI 推荐后用户调整）？
**推荐：** 后置。用户在输入产品后，AI 先根据内容策略推荐一个时长区间（比如"此产品适合 30-45s 展示"），用户可确认或调整。
**理由：** "先决策后生成"的 UX 在用户不了解产品能力时有压力。让 AI 先推荐，降低决策负担。

### 决策 4：明天上午是继续修复已知 bug，还是先跑通第一条真实端到端视频？
**推荐：** 先跑通第一条真实端到端视频。用真实的 API 产出 "Maternity Pillow" 30s 英文视频。如果成功，记录所有数据（耗时、API 调用次数、失败步骤）。如果失败，基于真实失败路径决定修复优先级。
**理由：** 我们今天的做法一直是"先修复所有可能出问题的地方再测试"——但这是瀑布思维。在敏捷里，你应该尽快拿到反馈。第一条视频的结果会告诉你哪些 bug 是真实的、哪些是想象中的。
